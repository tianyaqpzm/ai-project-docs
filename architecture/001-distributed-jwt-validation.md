# ADR 001: 分布式环境下的 JWT 本地化校验决策

## 状态
已通过 (Accepted)

## 背景
在当前的微服务体系中，`ms-java-gateway` 负责身份认证（SSO），但在下游服务（如 `ms-java-biz` 和 `ms-py-agent`）中缺乏统一的安全校验机制。如果攻击者绕过网关直接访问下游服务，系统将面临安全风险。

我们需要一种既能保证安全性，又不会显著增加内网调用延迟（例如避免每次请求都回查网关或认证中心）的方案。

## 决策
我们决定采用 **基于共享密钥的本地 JWT 校验方案**。

### 核心选型：
1.  **算法**：统一使用 **HMAC-SHA256 (HS256)** 对称加密算法。
2.  **密钥管理**：通过 Nacos 配置中心或环境变量 `JWT_SECRET` 共享 256 位密钥。
3.  **校验逻辑**：
    *   下游服务不再请求 Auth 服务，而是使用共享密钥自行验证签名。
    *   校验成功后，从 Payload 中提取 `sub` (User ID) 和角色信息。
4.  **Java 实现**：引入 `Spring Security` + `JJWT`。
5.  **Python 实现**：使用 `PyJWT` 手动实现 Fastapi 依赖注入。

## 备选方案与对比
*   **方案 A：网关集中校验 (Current)**
    *   *缺点*：内网服务处于“裸奔”状态，不符合零信任（Zero Trust）原则。
*   **方案 B：下游服务请求 Auth 服务验证 (Token Introspection)**
    *   *缺点*：增加网络开销和延迟，Auth 服务容易成为瓶颈。
*   **方案 C：本地 JWT 校验 (Chosen)**
    *   *优点*：性能极高，无网络开销；实现纵深防御；支持多语言栈。

## 后果
*   **密钥一致性**：必须保证集群内所有节点的 `JWT_SECRET` 严格一致，否则会导致校验失败。
*   **吊销机制**：由于是本地离线校验，不支持实时撤销 Token（可通过设置较短的过期时间或引入 Redis 黑名单解决，目前暂不实施）。
*   **开发成本**：每个新服务都需要配置基本的 Security 拦截层。

## 涉及组件
*   `ms-java-gateway` (Issuer)
*   `ms-java-biz` (Validator)
*   `ms-py-agent` (Validator)
