# ADR-006: 基于数据库触发器的聊天会话汇总同步方案

## 状态
已通过 (Accepted)

## 背景
在智能助手系统中，`chat_messages`（聊天消息表）是增长最快的数据表。当数据量达到百万级甚至千万级时，前端若直接通过 `GROUP BY session_id` 对消息表进行实时聚合来展示“会话列表”，会导致严重的查询性能问题和数据库负载压力。

此外，应用层手动维护汇总表（如在 Java 代码中同时更新消息和会话）会增加业务逻辑复杂度，且在多服务共享数据库或并发写入时难以保证最终一致性。

## 决策
1. **引入物理汇总表**：在 PostgreSQL 中建立 `chat_sessions` 表，专门用于存储每个会话的元数据（标题、最后活动时间等）。
2. **采用数据库触发器 (Trigger)**：在 `chat_messages` 表上创建 `AFTER INSERT` 触发器，由数据库底层自动完成数据的增量同步。
3. **优化同步逻辑**：
   - 消息插入时，自动更新 `chat_sessions` 的 `last_active_time`。
   - 若为该会话的第一条消息，自动创建会话记录并截取内容作为标题。

## 理由
1. **极速查询性能**：将会话列表的查询复杂度从 $O(N)$（全表聚合）降低到 $O(1)$（单表检索），支持前端秒级加载会话列表。
2. **应用层解耦**：符合本项目 [Java 开发规范](file:///Users/pei/projects/ms-java-biz/.agent/rules/ai-code-ws.md) 中关于“自动化同步”的要求。Java 业务代码只需关注消息持久化，无需关心汇总表状态。
3. **最终一致性**：利用数据库行级事务，确保消息产生与会话更新在物理层原子完成，避免了应用层可能出现的异常导致的数据不一致。
4. **降低网络负载**：减少了在大表上执行复杂聚合操作带来的 I/O 和 CPU 占用，特别是在远程数据库环境下尤为重要。

## 影响评估
- **写入开销**：每次插入消息会增加一次轻微的触发器执行和汇总表更新。在当前的 QPS 下，这种开销相对于性能收益几乎可以忽略不计。
- **存储开销**：增加了 `chat_sessions` 表的空间占用，但该表行数远少于消息表，属于合理的冗余设计。
- **迁移复杂度**：对于存量数据，需在迁移脚本（Flyway）中谨慎处理（建议采用异步迁移或分批处理以防超时）。
