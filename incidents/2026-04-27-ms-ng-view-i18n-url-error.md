# 问题复盘：ms-ng-view 国际化资源加载失败

**日期：** 2026-04-27
**状态：** 已解决
**严重性：** 高（导致应用无法正常初始化）

## 问题描述
用户反馈在 Angular 应用尝试加载国际化文件时出现 `Unknown Error (Status 0)`。控制台显示了一个畸形的 URL：`https://tao-lan.122577.xyz:8443./i18n/zh.json`。

## 根因分析
1. **拦截器范围过大**：`apiUrlInterceptor` 被配置为对所有不以 `http` 或 `/assets` 开头的请求拼接 `VITE_API_URL`。这错误地包含了以 `./i18n/` 开头的本地资源请求。
2. **字符串拼接错误**：逻辑中使用 `${baseUrl}${req.url}`。当 `baseUrl` 为 `...:8443` 且 `req.url` 为 `./i18n/...` 时，生成的字符串包含 `8443.`，这在 URL 的主机/端口部分是非法的，导致网络错误（Status 0）。
3. **目录拼写错误**：拦截器存储在名为 `intercepotors` 的目录中，违反了项目规范，且增加了调试和维护的难度。

## 解决方案
1. **限制拦截器范围**：更新了 `apiUrlInterceptor`，仅拦截并加前缀以斜杠 `/` 开头（类绝对路径）且不指向 `/assets` 的请求。现在拦截器会忽略以 `./` 开头的请求（用于 i18n 的相对路径），允许它们从本地加载。
2. **重构目录结构**：将 `src/app/core/intercepotors` 重命名为 `src/app/core/interceptors`（同步更新了 `tests/` 下的对应目录）。
3. **更新引用**：修复了 `main.ts` 和单元测试中所有指向更正后目录名的 import 语句。

## 经验教训
- **路径选择**：在拦截 API 调用时应使用更具体的标准（例如仅匹配以 `/` 开头的路径或特定的 API 前缀）。
- **静态资源处理**：托管在同一源下的静态资源通常应排除在修改 URL 的 API 拦截器之外。
- **命名规范**：定期检查核心目录名称是否存在拼写错误，因为它们会通过 import 语句在全工程扩散。

## 经验更新
已更新 `ms-ng-view/.agent/rules/ai-code-ws.md`，明确要求拦截器应小心处理以 `./` 开头的静态资源。
